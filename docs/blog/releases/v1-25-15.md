# v1.22.1 — v1.25.15 — Stabilization

<Badge type="tip" text="Stabilization Releases" />

**Release Range:** March 27 – July 12, 2026

Between v1.22.0 and v1.25.15, Goca shipped no new headline commands — instead, this stretch is five separate **stabilization passes**, each following the same method: generate real projects with every flag combination, actually run `go build`/`go vet`/`go test` against the output, and fix whatever broke, lied about success, or contradicted the docs. That method caught dozens of bugs that never surfaced in unit tests because they only appear in the *generated* code, not Goca's own.

This page consolidates all of it into one release note, the same way v1.19.0–v1.21.0 were consolidated into v1.22.0.

---

## Highlights

| Area | What changed |
|------|-------------|
| Custom templates | `goca template init` templates are now **actually used** by generation instead of being inert |
| Release pipeline | Fixed a silent CI bug where auto-created tags never triggered a real release for ~4 months |
| Multi-database support | Every generator now produces compiling code across all 8 supported databases, not just Postgres |
| DI / integration | `goca integrate`, `goca di --wire`, and multi-feature projects now wire correctly into `main.go` |
| Validation | DTO-layer validation now rejects bad input with 422 instead of leaking a 500 from the domain layer |
| Field handling | Custom field types, pointer (optional) fields, and camelCase field names all generate correctly now |
| Docs | Large coherence pass across `docs/commands/*.md` and the wiki, fixing broken examples and stale "generated code" samples |

## Five Stabilization Batches

Each batch generated real projects across every database backend and handler type, compiled the output, and fixed what broke:

- **Batch 1** — ~10 bugs across entities, CLI handlers, mocks, DI, and config: deduplicated colliding error constants, fixed use-case constructor naming for multi-word entities (`NewUserProfileService`, not `NewUserprofileService`), and more.
- **Batch 2** — `goca di --wire`'s generated container type collided with the regular one (`Container` declared twice); rewrote `--integration-tests` output, which had drifted entirely from the real use-case/repository API (no `ctx` parameter, `Get<Entity>(int(id))`, `List<Entity>s()` returning `.Total`).
- **Batch 3** — ~60 bugs across every command: field names normalized to idiomatic PascalCase while preserving snake_case JSON/GORM tags, stub types generated for custom field types instead of emitting undefined identifiers, and a full audit verifying `goca init` + 2 features + a tricky-field entity compiles cleanly end-to-end with DI wired, both feature route groups served, and both entities auto-migrating.
- **Batch 4** — repository constructor wiring was hardcoded to `NewPostgres<Entity>Repository` regardless of database; `repoConstructorPrefix(database)` now makes DI/feature/integrate/tests reference the right constructor for every backend (sqlite, postgres, mysql, postgres-json, sqlserver, mongodb).
- **Batch 5** — the gRPC handler scaffold broke `go vet`/`go mod tidy` entirely (it imported a `pb` package that's only generated by `protoc`, so tooling attempted a remote lookup and failed) — now generates a build-tagged placeholder package so the import resolves locally; `--middleware-types` generated a middleware package that routes.go never actually imported.

Two more targeted fixes in the same spirit:
- **DB connectivity**: a fresh SQLite project (the default database) never actually connected — `GetDatabaseURL()` always emitted a Postgres DSN regardless of driver, and a "development + empty password → degraded mode" guard meant to help Postgres blocked SQLite (which has no password) from connecting at all. Fixed per-driver, and entities are now registered in `runAutoMigrations` so tables actually exist.
- **`goca integrate --all`** claimed routes were registered but every endpoint 404'd — the complete-rewrite `main.go` path had the same template defects as `init`'s, *and* was blocked from ever overwriting the existing file by the safety guard it was supposed to go through. Now extends the real init-generated `main.go` in place.
- **Email validation** returned HTTP 500 for a malformed email instead of 422 — the DTO `validate` tag was type-only (`required,min=1`); it's now field-name-aware, so email fields get `required,email` and the HTTP layer rejects bad input before it ever reaches the domain layer.

## Custom Templates Actually Work Now

`goca template init` scaffolds `.goca/templates/*.tmpl`, but no generator ever called `ExecuteCustomTemplate` on them — editing a template did nothing, despite the CLI's own messaging claiming otherwise. `domain/entity`, `usecase/dto`, `handler/http/handler` and `repository/repo` are now wired into `goca entity`/`goca feature`/`goca handler`/`goca repository`, falling back to the built-in generator when no custom template exists (or, for the two merge-based files, after the first entity in a project). The shipped default templates — which had never actually been executed — were also full of bugs: a missing `gorm.io/gorm` import, an unconditional `"time"` import, a malformed method receiver, and DTO/handler templates whose API (`...Request`/`...Response`, `Create(ctx, req)`) never matched what the real generator produces.

Wiring this up surfaced a second bug: loading `.goca.yaml` (which every generate command already does) was silently auto-creating `.goca/templates/` with the unmodified defaults on first use — which the new wiring would then treat as deliberate customization, activating custom generation for *every* project instead of just ones that opted in via `goca template init`. Fixed by splitting template *loading* (read-only) from template *initialization* (explicit, opt-in).

## Release Pipeline Fix

`auto-release.yml` creates a tag and pushes it using the default `GITHUB_TOKEN` — but GitHub's anti-recursion protection silently prevents a `GITHUB_TOKEN`-authored push from triggering another workflow. That meant `release.yml`'s `push: tags: v*` trigger never fired: every auto-created tag since v1.22.1 (17 of them) created a GitHub tag but never built binaries or published a release. `auto-release.yml` now explicitly dispatches `release.yml` via the API (`workflow_dispatch`, which isn't subject to that restriction) right after tagging. `release.yml` was also still pinned to Go 1.21 while `go.mod` requires 1.25.1+.

## Generator Bugs Found by an Exhaustive Flag Audit

A follow-up pass validated every flag on all 18 commands against real generated output and fixed what it found:

- `goca feature --dry-run` genuinely wrote `cmd/server/main.go` (auto-migration registration) despite claiming to preview only — broke the build the moment you inspected the result.
- A field with a custom type (`sku:CustomType`) produced an unqualified `CustomType` reference instead of `domain.CustomType` outside the domain package.
- Pointer fields (`*string`, `*time.Time` — the documented way to make a field optional) got `gorm:"not null"`, contradicting the whole point of a pointer.
- camelCase field names (`createdAt`, `userID`) lost their internal capitalization (`createdAt` → `Createdat`) — snake_case input was unaffected.
- `--backup` alone could never actually overwrite an existing file, contradicting its own error message.
- `goca config init --template` fabricated a placeholder module path instead of reading the real one from `go.mod`.
- A module-name/entity-name collision (e.g. project `bookstore` + entity `Book`) silently dropped that entity's HTTP routes from `main.go`.

## Documentation

A full coherence pass across `docs/commands/*.md` and the GitHub wiki fixed broken examples (`goca integrate --feature` doesn't exist — it's `-f/--features`), several "generated code" samples that had gone stale as the real generator evolved (shown `context.Context`/`database/sql`/`...Request`/`...Response` APIs that haven't been accurate for a while), missing flags (`--wire` on `di`, `--async` on `usecase`, the `--all`/`--repository`/`--usecase`/`--handler` selectors on `mocks`), and wrong defaults in several wiki pages that flatly contradicted `--help`.

---

## Known Gaps (documented, not yet fixed)

- `goca config init` and `goca init --config` generate two different `.goca.yaml` shapes via independent code paths — the former is missing the `templates`/`testing`/`features`/`deploy` sections, so `goca template init` silently does nothing useful against it.
- Any entity name starting with "W" (Widget, Wallet, Warehouse...) breaks HTTP handler generation: the handler receiver `w *WidgetHandler` collides with the `w http.ResponseWriter` parameter name.

See [CHANGELOG.md](https://github.com/sazardev/goca/blob/master/CHANGELOG.md) for the complete, commit-level history.

## Upgrading

```bash
go install github.com/sazardev/goca@latest
goca version
```

No breaking changes across this range — all fixes are corrections to existing generator output.
