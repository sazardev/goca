# Project Structure

Understanding the directory organization and file conventions.

## Overview

Goca follows Clean Architecture principles with clear separation of concerns. This is what `goca init` actually scaffolds:

```
myproject/
├── cmd/
│   └── server/
│       └── main.go        # Application entry point
├── internal/               # Private application code
│   ├── domain/            # Business entities
│   ├── usecase/           # Business logic
│   ├── repository/        # Data access
│   └── handler/           # Delivery mechanisms (HTTP handlers, routes)
├── pkg/                    # Shared packages (created by init, not project-public libs)
│   ├── config/
│   │   └── config.go       # Config struct + Load() — plain Go, not YAML
│   ├── logger/
│   │   └── logger.go
│   └── auth/                # Only with --auth
│       └── jwt.go
├── migrations/
│   ├── 001_initial.up.sql
│   ├── 001_initial.down.sql
│   └── README.md
├── .goca.yaml               # Project configuration (see Configuration Guide)
├── go.mod
└── go.sum
```

`internal/di/` (dependency injection container) and `internal/messages/` (typed errors/response messages) are **not** created by `goca init` — they appear once you run `goca di` or `goca feature`/`goca messages`, which generate into the project on top of this base structure.

## Layer Responsibilities

### Domain (`internal/domain/`)

Contains business entities and core business rules.

**Files:**
- `{entity}.go` - Entity definition
- `{entity}_seeds.go` - Seed data (when `--seeds` is used)
- `errors.go` - Domain-specific errors

**Example:**
```go
// user.go
package domain

type User struct {
    ID        uint   `gorm:"primaryKey"`
    Name      string `gorm:"not null"`
    Email     string `gorm:"unique;not null"`
    CreatedAt time.Time
    UpdatedAt time.Time
}
```

### Use Case (`internal/usecase/`)

Business logic and application rules.

**Files:**
- `{entity}_service.go` - Business logic implementation
- `interfaces.go` - Service contracts
- `dto.go` - Data transfer objects

**Example:**
```go
// user_service.go
package usecase

type UserService interface {
    CreateUser(ctx context.Context, input CreateUserInput) (*UserResponse, error)
    GetUser(ctx context.Context, id uint) (*UserResponse, error)
}
```

### Repository (`internal/repository/`)

Data persistence layer. The implementation file is named after the database driver:

| Database(s)                                | Generated file                          |
| ------------------------------------------- | ---------------------------------------- |
| `postgres`, `mysql`, `sqlite`                | `postgres_{entity}_repository.go` (shared implementation — all three run on GORM) |
| `postgres-json`                              | `postgres_json_{entity}_repository.go`   |
| `sqlserver`                                  | `sqlserver_{entity}_repository.go`       |
| `mongodb`                                    | `mongo_{entity}_repository.go`           |
| `dynamodb`                                   | `dynamodb_{entity}_repository.go`        |
| `elasticsearch`                              | `elasticsearch_{entity}_repository.go`   |

Plus, always: `interfaces.go` (repository contracts). With `--cache`: `cached_{entity}_repository.go`. With `goca mocks`: `mock_{entity}_repository.go`.

**Example:**
```go
// postgres_user_repository.go
package repository

type PostgresUserRepository struct {
    db *gorm.DB
}

func (r *PostgresUserRepository) Save(user *domain.User) error {
    return r.db.Create(user).Error
}
```

### Handler (`internal/handler/`)

HTTP delivery layer, built on **gorilla/mux** (not gin/chi/echo).

**Files:**
- `{entity}_handler.go` - HTTP handlers
- `routes.go` - Route registration (`Setup{Entity}Routes(router *mux.Router, ...)`)

**Example:**
```go
// user_handler.go
package handler

type UserHandler struct {
    service usecase.UserService
}

func (h *UserHandler) CreateUser(w http.ResponseWriter, r *http.Request) {
    // mux.Vars(r) for path parameters
}
```

### Messages (`internal/messages/`)

Generated by `goca messages` (or as part of `goca feature`): typed error constants and response message structs shared by handlers and use cases.

### Dependency Injection (`internal/di/`)

Generated by `goca di` (or as part of `goca feature`): wires all components together.

**Files:**
- `container.go` - DI container (manual wiring by default)
- `wire.go`, `wire_container.go` - only when generated with `--wire`

**Example:**
```go
// container.go
package di

type Container struct {
    UserRepository repository.UserRepository
    UserService    usecase.UserService
    UserHandler    *handler.UserHandler
}
```

## File Naming Conventions

| Pattern                           | Example                       | Purpose           |
| --------------------------------- | ----------------------------- | ------------------ |
| `{entity}.go`                     | `user.go`                     | Entity definition  |
| `{entity}_service.go`             | `user_service.go`             | Business logic     |
| `postgres_{entity}_repository.go` | `postgres_user_repository.go` | Data access (postgres/mysql/sqlite) |
| `mongo_{entity}_repository.go`    | `mongo_user_repository.go`    | Data access (mongodb) |
| `{entity}_handler.go`             | `user_handler.go`             | HTTP handlers      |
| `{entity}_seeds.go`               | `user_seeds.go`               | Seed data          |
| `{entity}_test.go`                | `user_test.go`                | Unit tests         |

## Package Organization

### Internal vs Pkg

```
internal/     # Private to this application — Clean Architecture layers
pkg/          # Shared support packages goca init scaffolds (config, logger, auth)
```

### Feature Grouping

Each feature spans multiple layers:

```
User Feature:
├── internal/domain/user.go
├── internal/usecase/user_service.go
├── internal/repository/postgres_user_repository.go
└── internal/handler/user_handler.go
```

## Import Rules

Follow dependency direction:

```
Handler → Use Case → Repository → Domain
   ↓         ↓           ↓          ↓
 HTTP    Business    Database   Entities
```

**Allowed:**
```go
// Handler imports use case
import "myproject/internal/usecase"

// Use case imports repository interface
import "myproject/internal/repository"
```

**Not Allowed:**
```go
// Repository imports handler (wrong direction!)
import "myproject/internal/handler"
```

`goca analyze --arch` checks this rule automatically.

## Configuration Files

Goca does not generate a `config/` directory of YAML files. Configuration lives in two places:

```
.goca.yaml          # Goca's own project configuration (database, naming, features — see the Configuration Guide)
pkg/config/config.go # Application runtime config: a Go struct + Load() function, typically populated from environment variables
```

## Migrations

```
migrations/
├── 001_initial.up.sql
├── 001_initial.down.sql
└── README.md
```

Migration files use a zero-padded sequence number (`001`, `002`, ...), not a timestamp prefix.

## Testing Structure

Mirror the source structure:

```
internal/
├── usecase/
│   ├── user_service.go
│   └── user_service_test.go
└── repository/
    ├── postgres_user_repository.go
    └── postgres_user_repository_test.go
```
