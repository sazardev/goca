# goca analyze Command

The `goca analyze` command performs a comprehensive **deep static self-analysis** of the project generated by Goca. It inspects the content of generated Go code across 6 categories and 30 rules.

## 📋 Syntax

```bash
goca analyze [flags]
```

## 🎯 Purpose vs. `goca doctor`

| | `goca doctor` | `goca analyze` |
|---|---|---|
| Checks | Structural (dirs, build, vet) | Content (AST, imports, patterns) |
| Depth | Shallow (6 checks) | Deep (30 rules across 6 categories) |
| Security | None | OWASP A03 patterns, unsafe, TLS |
| Architecture | Dir presence only | Import direction enforcement |
| Tests | None | Table-driven, mocks, tempdir |

## 🚩 Available Flags

| Flag             | Type     | Default  | Description                                              |
| ---------------- | -------- | -------- | -------------------------------------------------------- |
| `--arch`         | `bool`   | `false`  | Only run architecture checks                             |
| `--quality`      | `bool`   | `false`  | Only run code quality checks                             |
| `--security`     | `bool`   | `false`  | Only run security checks                                 |
| `--standards`    | `bool`   | `false`  | Only run Go standards checks                             |
| `--tests`        | `bool`   | `false`  | Only run test quality checks                             |
| `--deps`         | `bool`   | `false`  | Only run dependency checks                               |
| `--output`       | `string` | `text`   | Output format: `text` or `json`                          |
| `--fail-on-warn` | `bool`   | `false`  | Exit with code 2 when warnings are present               |

When no category flag is set, **all 6 categories** run.

## 📖 Usage Examples

### Full deep analysis
```bash
goca analyze
```

### Security-only
```bash
goca analyze --security
```

### Strict CI (fail on warnings too)
```bash
goca analyze --fail-on-warn
```

### JSON output for tooling
```bash
goca analyze --output json
```

## 🔍 Check Categories

### Architecture (6 rules)
- `layer-dirs` — All 4 Clean Architecture layer directories exist
- `domain-purity` — `internal/domain` imports only stdlib (no ORMs)
- `usecase-no-handler` — Use cases do not import the handler layer
- `repo-impl-coverage` — Each domain entity has a repository implementation
- `di-container` — DI container exists in `internal/di`
- `handler-no-repo` — Handlers do not import repositories directly

### Quality (5 rules)
- `no-empty-files` — No effectively empty `.go` files
- `package-naming` — Packages are lowercase with no underscores
- `no-todos` — No `// TODO` or `// FIXME` in generated code
- `exported-docs` — All exported functions have godoc comments
- `main-go` — An entry point `main.go` exists

### Security (5 rules)
- `no-hardcoded-secrets` — No hardcoded passwords/tokens/secrets
- `no-sql-injection` — No `fmt.Sprintf` SQL construction (OWASP A03)
- `no-unsafe` — No `import "unsafe"` in application code
- `no-tls-skip` — No `InsecureSkipVerify: true`
- `env-config` — Sensitive config uses `os.Getenv()`

### Standards (5 rules)
- `file-naming` — Files use `snake_case` (not kebab-case)
- `no-init-in-domain` — No `init()` in domain layer
- `go-mod-valid` — `go.mod` has valid module declaration
- `goca-yaml` — `.goca.yaml` is present and configured
- `context-propagation` — `context.Context` used in I/O layers

### Tests (4 rules)
- `test-files-exist` — Each non-empty layer has `*_test.go` files
- `table-driven-tests` — Table-driven tests (`[]struct{...}`) detected
- `mocks-exist` — Mock directory is present
- `test-tempdir` — Tests use `t.TempDir()`, not `/tmp`

### Dependencies (4 rules)
- `go-mod-tidy` — `go.sum` present when `go.mod` has `require`
- `no-replace` — No `replace` directives
- `go-version` — Go version declared in `go.mod`
- `no-insecure-modules` — No deprecated/insecure modules (e.g., `dgrijalva/jwt-go`)

## 📊 Exit Codes

| Code | Meaning |
|------|---------|
| `0` | All checks passed |
| `1` | One or more checks failed |
| `2` | Warnings present with `--fail-on-warn` |
